## 1부: 색공간의 이해

### 색공간이란

색공간(color space)은 **색을 숫자로 표현하는 약속**입니다. "빨간색"이라는 말은 모호하지만, `rgb(255, 0, 0)` 또는 `oklch(0.628, 0.258, 29.23)`은 정확한 하나의 색을 가리킵니다. 각 색공간은 서로 다른 축(좌표계)으로 색을 기술하며, 같은 색이라도 표현 방식이 다릅니다.

### sRGB: 웹의 기본 색공간

sRGB(standard RGB)는 1996년에 HP와 Microsoft가 공동 제정한 색공간으로, 모니터, 프린터, 디지털 카메라 등 소비자 기기의 공통 언어가 되었습니다. 웹에서 `#ff0000`, `rgb(255, 0, 0)`, `hsl(0, 100%, 50%)`은 모두 sRGB 좌표계로 빨간색을 표현합니다.

sRGB는 인간이 볼 수 있는 모든 색의 일부만 표현할 수 있습니다. 이 "표현 가능한 범위"를 **가뭇(gamut)**이라 하며, sRGB 가뭇은 상당히 좁습니다. Display P3나 Rec. 2020 같은 넓은 가뭇의 디스플레이가 보급되면서, CSS도 sRGB를 넘어선 색공간을 지원하게 되었습니다.

### CIE XYZ: 모든 색공간의 교환 공간

1931년 CIE(국제조명위원회)는 인간 시각 실험에 기반하여 **XYZ 색공간**을 정의했습니다. XYZ의 핵심 특징:

- **인간이 볼 수 있는 모든 색**을 양수 좌표로 표현할 수 있습니다.
- Y 축은 밝기(luminance)에 대응합니다. WCAG의 상대 휘도 계산이 이 Y값을 사용합니다.
- 거의 모든 색공간 변환은 **XYZ를 경유**합니다: `색공간A → XYZ → 색공간B`.

XYZ는 특정 기기에 종속되지 않는 보편 좌표계이므로, 서로 다른 색공간 사이의 "번역 허브" 역할을 합니다.

### D50 vs D65: 흰색의 기준이 다르다

색을 정의하려면 "흰색이 무엇인가"를 먼저 정해야 합니다. 이 기준 흰색을 **백색점(white point)**이라 하고, 표준 광원(illuminant)으로 정의합니다.

| 광원 | 색온도 | 용도 | 설명 |
|------|--------|------|------|
| D50 | ~5000K | 인쇄, CIE LAB | 맑은 날 수평선 근처의 햇빛. 약간 따뜻한 흰색 |
| D65 | ~6500K | 모니터, sRGB, OKLAB | 흐린 날 북쪽 하늘. 약간 차가운 흰색 |

CIE LAB은 역사적으로 D50을 기준으로 정의되었고, sRGB와 OKLAB은 D65를 기준으로 합니다. 따라서 **LAB → sRGB 변환에는 D50 → D65 색순응(chromatic adaptation)이 필수**입니다.

## 2부: LAB/LCH — 인간 지각 기반 색공간

### CIE LAB

CIE LAB(또는 CIELAB, L\*a\*b\*)은 1976년에 정의된 색공간으로, 세 축이 인간의 색 지각에 대응합니다:

- **L** (Lightness, 0~100): 밝기. 0은 검정, 100은 흰색.
- **a** (−128 ~ +128): 초록(−) ↔ 빨강(+) 축
- **b** (−128 ~ +128): 파랑(−) ↔ 노랑(+) 축

LAB의 핵심 설계 목표는 **지각적 균등성(perceptual uniformity)**입니다. 좌표 거리가 같으면 사람이 느끼는 색 차이도 비슷해야 합니다. 실제로는 파란색 영역에서 이 균등성이 깨지지만, sRGB보다는 훨씬 균등합니다.

### CIE LCH

LCH는 LAB의 **극좌표(polar) 표현**입니다:

- **L**: LAB과 동일한 밝기
- **C** (Chroma): 채도. `sqrt(a² + b²)` — 원점에서의 거리
- **H** (Hue): 색상각. `atan2(b, a)`를 도(degree)로 표현

직교좌표(a, b)보다 극좌표(C, H)가 색상을 직관적으로 조작하기에 유리합니다. "같은 밝기에서 색상만 바꾸기"는 H만 회전시키면 됩니다.

**변환 수식 (LCH → LAB):**

```text
a = C * cos(H * π/180)
b = C * sin(H * π/180)
L은 그대로 전달
```

### LAB → XYZ: 구간별 역함수

LAB에서 XYZ를 복원하는 과정은 LAB의 정의를 역으로 풀어야 합니다. LAB 정의에는 **구간별(piecewise) 함수**가 있습니다:

```text
f(t) = t^(1/3)              if t > ε
f(t) = (κt + 16) / 116      if t ≤ ε
```

여기서 `ε = 216/24389 ≈ 0.008856`, `κ = 24389/27 ≈ 903.296`.

이 두 구간이 필요한 이유: 세제곱근 함수(`t^(1/3)`)는 0 근처에서 기울기가 무한대로 발산합니다. 아주 어두운 색에서 미세한 XYZ 차이가 LAB 좌표의 거대한 변화로 증폭되는 것을 막기 위해, 어두운 영역(`t ≤ ε`)에서는 **선형 근사**로 대체합니다.

역변환은 이 과정을 뒤집습니다:

1. **f값 복원**: `fy = (L+16)/116`, `fx = a/500 + fy`, `fz = fy - b/200`
2. **구간별 역변환**:
   - x: `fx³ > ε`이면 `xr = fx³`, 아니면 `xr = (116·fx - 16) / κ`
   - y: `L > κ·ε`이면 `yr = ((L+16)/116)³`, 아니면 `yr = L / κ`
   - z: `fz³ > ε`이면 `zr = fz³`, 아니면 `zr = (116·fz - 16) / κ`
3. **D50 백색점 스케일링**: `X = xr × D50.X`, `Y = yr × D50.Y`, `Z = zr × D50.Z`

구현: [`src/convert/lab-to-xyz.ts`](../../src/convert/lab-to-xyz.ts)

### Bradford 색순응: D50 → D65

LAB에서 얻은 XYZ는 D50 기준입니다. sRGB 변환을 위해 D65 기준으로 바꿔야 합니다. 단순히 백색점 비율로 스케일링하는 것이 아니라, **Bradford 색순응 변환**이라는 3×3 행렬을 사용합니다.

Bradford 모델은 인간의 시각 적응을 모사합니다. 실내에서 밖으로 나가면 조명 색온도가 바뀌지만, 우리 눈은 적응하여 흰 종이를 여전히 "희다"고 인지합니다. Bradford 행렬은 이 적응 과정을 수학적으로 표현합니다.

`labToXyz` 함수는 이 Bradford 변환을 내부에 캡슐화하여, 호출자는 항상 D65 기준의 XYZ를 받습니다.

## 3부: OKLAB/OKLCH — 현대적 대안

### OKLAB이 등장한 배경

CIE LAB은 50년 가까이 사용되었지만, 알려진 문제가 있습니다:

- **파란색 영역의 색상 왜곡**: LAB에서 밝기만 바꾸면 파란색이 보라색으로 shift됩니다.
- **채도 균등성 부족**: LAB의 "동일 거리 = 동일 지각 차이" 원칙이 모든 영역에서 성립하지 않습니다.

2020년 Björn Ottosson이 발표한 **OKLAB**은 이 문제들을 해결한 현대적 지각 균등 색공간입니다.

### OKLAB → XYZ: LMS를 경유하는 이유

OKLAB 변환은 인간 망막의 **세 가지 원뿔 세포(cone cell)**를 모델링합니다:

- **L cone**: 긴 파장(빨강 계열) 감지
- **M cone**: 중간 파장(초록 계열) 감지
- **S cone**: 짧은 파장(파랑 계열) 감지

이 세 세포의 응답을 **LMS 색공간**이라 합니다. OKLAB은 LMS의 세제곱근에서 정의되므로, 역변환은:

1. **OKLAB → LMS (세제곱근 도메인)**: 3×3 행렬 곱셈
2. **세제곱**: 각 성분을 세제곱하여 선형 LMS로 복원
3. **LMS → XYZ-D65**: 3×3 행렬 곱셈

OKLAB은 처음부터 D65를 기준으로 설계되었으므로, LAB과 달리 **색순응 변환이 필요 없습니다**.

구현: [`src/convert/oklab-to-xyz.ts`](../../src/convert/oklab-to-xyz.ts)

### OKLCH: CSS 디자인 시스템에서의 인기

OKLCH는 OKLAB의 극좌표 표현이며, LCH가 LAB의 극좌표인 것과 같은 관계입니다. CSS 디자인 시스템에서 OKLCH가 인기를 얻은 이유:

- **채도 축의 독립적 조작**: 색상(H)을 바꿔도 지각적 밝기(L)가 유지됩니다.
- **일관된 색상 팔레트 생성**: L과 C를 고정하고 H만 회전시키면, 시각적으로 균형 잡힌 팔레트가 만들어집니다.
- **LAB보다 정확한 지각 균등성**: 파란색 영역에서도 예측 가능하게 동작합니다.

변환 수식은 LCH → LAB과 동일합니다:

```text
a = C * cos(H * π/180)
b = C * sin(H * π/180)
```

구현: [`src/convert/oklch-to-oklab.ts`](../../src/convert/oklch-to-oklab.ts)

## 4부: XYZ → Linear sRGB

### 3×3 행렬 곱셈의 기하학적 의미

XYZ와 linear sRGB는 모두 선형 색공간입니다. 둘 사이의 변환은 **좌표축 회전(및 스케일링)**에 해당합니다. 3×3 행렬 한 번의 곱셈으로 완료됩니다:

```text
[r]   [M00 M01 M02] [x]
[g] = [M10 M11 M12] [y]
[b]   [M20 M21 M22] [z]
```

이 행렬은 sRGB의 원색(primaries) 좌표와 D65 백색점에서 유도됩니다.

### 유리수 행렬이 중요한 이유

CSS Color Level 4 스펙은 이 행렬을 **유리수(분수)**로 제공합니다:

```text
12831/3959  대신  3.2404541621...
```

부동소수점 연산에서 소수점 이하 자릿수가 많아질수록 반올림 오차가 누적됩니다. 유리수를 JavaScript의 나눗셈(`12831 / 3959`)으로 직접 계산하면, IEEE 754 배정밀도 부동소수점이 허용하는 최대 정밀도를 얻을 수 있습니다.

구현: [`src/convert/xyz-to-linear-rgb.ts`](../../src/convert/xyz-to-linear-rgb.ts)

### 범위를 벗어나는 값

`xyzToLinearRgb`의 출력은 [0, 1] 범위를 벗어날 수 있습니다. 예를 들어 `oklch(90% 0.4 150)`처럼 sRGB 가뭇 밖의 색을 변환하면, r/g/b 중 일부가 음수이거나 1보다 클 수 있습니다.

이것은 **올바른 동작**입니다. sRGB가 표현할 수 없는 색이 실제로 존재하며, 변환 함수는 이 사실을 정직하게 보고합니다. 범위 밖 값의 처리는 별도의 가뭇 매핑 단계에서 담당합니다.

## 5부: 가뭇 매핑 — 범위 밖 색상 처리

### 가뭇(gamut)이란

가뭇은 특정 색공간이 **표현할 수 있는 색의 범위**입니다. sRGB의 가뭇은 Display P3보다 작고, Display P3은 Rec. 2020보다 작습니다. CIE XYZ와 OKLAB은 이론적으로 가뭇 제한이 없는 색공간입니다.

CSS에서 `oklch(90% 0.4 150)`을 선언하면, 이 색은 sRGB 가뭇 밖에 있을 수 있습니다. 이때 브라우저는 어떻게 할까요?

### 세 가지 선택지

1. **클리핑(clipping)**: 각 채널을 [0, 1]로 잘라냅니다. 간단하지만, 색상과 밝기가 크게 왜곡될 수 있습니다.
2. **거부(rejection)**: 에러를 발생시킵니다. 하지만 브라우저에서 렌더링되는 유효한 CSS 색상을 거부하는 것은 부적절합니다.
3. **CSS 가뭇 매핑**: 지각적으로 가장 가까운 sRGB 색을 찾습니다.

### CSS Color Level 4 가뭇 매핑 알고리즘

CSS 스펙이 정의한 가뭇 매핑 알고리즘의 핵심 아이디어:

> OKLCH에서 **채도(C)를 줄여가며** sRGB 경계 위의 가장 가까운 색을 찾는다.

왜 OKLCH에서 채도를 줄이는가? OKLCH는 지각 균등 색공간이므로, 채도만 줄이면 **밝기와 색상은 보존**하면서 sRGB 가뭇 안으로 들어올 수 있습니다. 이는 지각적으로 원래 의도에 가장 가까운 색을 만듭니다.

**알고리즘 요약:**

1. 변환 대상 색을 OKLCH로 변환합니다.
2. L(밝기)이 0 이하면 검정, 1 이상이면 흰색을 반환합니다.
3. C(채도)를 0으로 줄이면 sRGB 안에 들어오므로, `C_min=0`과 `C_max=현재C` 사이에서 **이진 탐색**합니다.
4. 각 반복에서:
   - 중간 채도로 OKLCH → sRGB 변환
   - sRGB 범위 내인지 확인
   - 범위 내이면 C_min을 올리고, 밖이면 C_max를 내림
5. 수렴 조건: **deltaEOK < 0.02** (OKLAB 색공간에서의 지각적 색차)
6. 안전 장치: **최대 64회 반복**

`deltaEOK < 0.02`는 "인간이 차이를 거의 감지할 수 없는 수준"을 의미합니다. 64회 반복 제한은 무한 루프 방지용이지만, 실제로는 훨씬 적은 반복으로 수렴합니다.

### 이 도구에서의 적용

이 도구는 WCAG 명도대비를 계산합니다. 사용자가 `oklch(90% 0.4 150)`으로 텍스트 색상을 지정하면, **브라우저가 실제로 렌더링하는 sRGB 색**과 동일한 색으로 대비를 계산해야 합니다. 그래야 "화면에서 사용자가 실제로 보는 것"에 대한 정확한 접근성 평가가 가능합니다.

이 결정의 자세한 근거는 [ADR-002](../decisions/002-gamut-mapping.md)에 기록되어 있습니다.

## 변환 체인 요약

```text
LAB 계열:
  LCH ──→ LAB ──→ XYZ-D50 ──Bradford──→ XYZ-D65 ──→ Linear sRGB
  극좌표    직교좌표   구간별역함수        색순응         행렬곱셈

OKLAB 계열:
  OKLCH ──→ OKLAB ──→ LMS^(1/3) ──cube──→ LMS ──→ XYZ-D65 ──→ Linear sRGB
  극좌표     직교좌표    행렬곱셈     세제곱    행렬곱셈          행렬곱셈
```

두 경로 모두 **XYZ-D65에서 합류**하여 같은 `xyzToLinearRgb` 함수를 공유합니다.
